## 候选 RQ4：价值函数设计 / 训练信号（What is the Value Function?）

> **研究问题**：从 raw reward 到优化目标，价值函数是如何设计的？不同训练信号对记忆策略的学习有什么影响？

### 背景

当前大多数工作使用 QA accuracy 作为训练信号，但 Spurious Rewards 论文表明完全随机奖励也能取得接近真实奖励的效果，暗示我们甚至不确定目前的训练信号到底在教模型什么。

### 已发现的训练信号类型

|训练信号类型|代表论文|信号来源|
|---|---|---|
|QA accuracy|Memory-R1, MemAgent, MemSearcher|下游答案正确性|
|QA + compression|Mem-α|答案正确 + 惩罚冗余存储|
|Self-certainty|RLIF|模型自身对答案的置信度|
|Hidden state matching|Knowledge Modules|Teacher 模型隐藏状态|
|Random / spurious|Spurious Rewards|完全随机|
|Execution result|Memory as Action, Scaling Context Folding|可执行验证（对应 RQ3 G5）|

### 待明确

- 定义 reward 到优化目标 / 价值函数的映射与计算流程
- 分类准则与评估方法
- 可能与 RQ3（优化目标）及"非 QA 训练范式"方向合并
- 与 RQ4（RL 算法）的边界：候选 RQ4 关注"训练信号来自哪里"，RQ4 关注"用什么 RL 算法把信号转成梯度"。

## 候选 RQ5：记忆操作机制（How are Memory-related Actions Conducted?）

> **研究问题**：在推理阶段，记忆的读 / 写 / 存 / 取 / 用是如何被触发和执行的？谁负责执行？策略是什么？

### 按记忆类型分析

**外部存储 / 检索型记忆（T1, T2）**：

- 谁负责读 / 写 / 存？（主模型、辅助模块、规则引擎？）
- 写入策略：什么触发方式引发写入（定时 / 语义阈值 / LLM 判断 / 工具调用）？写入什么粒度？（原文 / 摘要 / 结构化条目）
- 读取策略：什么触发方式引发检索？检索的 query 如何构造？
- 更新策略：如何合并、去重、遗忘？

**Context window 型记忆（T3）**：

- 在固定 budget 下，如何决定保留什么、丢弃什么？
- 压缩策略：提取式 vs 生成式 vs 混合
- 读写的定义：在 context window 内如何区分"读"和"用"？

**参数化 / 隐式记忆（T4, T5, T6）**：

- 可编辑性：能否针对特定记忆条目进行修改？
- 触发机制：推理时记忆如何被激活（gate / attention routing / 条件 prompt）？

> 记忆的执行机制目前先作为 Inference 部分的笔记条目存在，等整理出明确的分类维度后再升级。

## 开放问题（Open Questions，未归入任何 RQ）

### Q-Explicit-vs-Param：显式 vs 隐式 知识边界
 
> **问题**：何种知识 / 信息适合存入**显式记忆**（T1–T3），何种更适合被作为**隐式记忆**（T4-T5）？

**动机**：

- 显式记忆：可解释、可编辑、推理时需要检索；访问成本与存储规模成正比。
- 参数化记忆：推理时"零成本"访问，但更新成本高、可能灾难性遗忘、不可解释。
- 实践中两种存储的边界并不清晰，常识性知识是否应该全部参数化？个体用户画像是否必须显式？

**可能的判定维度**：

1. 更新频率（高频 - 显式，低频 - 参数化）
2. 个性化程度（个体相关 - 显式，通用 - 参数化）
3. 事实 vs 技能（事实 - 显式，技能 - 参数化）
4. 解释性需求（需要解释 - 显式）
5. 规模（条目数量、总信息量）

**初步答案**：

- **适合参数化（隐式）**：塑造"我是谁"的信息：用户身份、偏好、常用习惯。这类信息访问频率极高、对延迟敏感、可容忍一定模糊性，适合以极高信息密度固化到权重中作为"随身的隐式记忆"。
- **适合显式存储**：访问频率极低、不常查、但查的时候必须精准且可回溯的信息（如具体事实、历史记录、可验证的引用来源）。因为参数化记忆本质上是概率性的，会有模糊感，不适合需要精确召回的场景。
- 需继续尝试用一个更完整的框架进行分类。

### Q-Distillation：模型蒸馏在 memory 语境下的意义

> **问题**：模型蒸馏（distillation）它到底会帮助我在哪个目标上有所提升？它与 plug-in 模块、全参数重训相比有什么独特价值？对 memory 和我们有什么意义？

#### 1. 蒸馏于 RQ1 而言：蒸馏即"显式 → 隐式"的内化过程

传统认知中，学生模型的能力不应超过教师模型。但实证研究表明并非如此：教师模型输出比原始数据集信息密度更高，使学生模型收敛更快、数据需求更少，最终性能常常反超同规模从零训练的模型。

用 RQ1 的定义来表达：

蒸馏是一个将"源自过去"的复杂交互轨迹，通过教师模型的理解进行提炼，并最终以极高的信息密度固化到学生模型参数中的过程。

这正是 RQ1 中"内化为隐式记忆（T4）"的过程。同时它也是一种参数化的手段。换句话说，蒸馏为 memory 提供了一个可学习、可压缩、可插拔的能力载体。

#### 2. 蒸馏于 RQ2 而言：蒸馏可作用于目前已知的所有组件的优化

蒸馏可以作用于 RQ2 中列出的任何一个组件（O1–O5）：

这些辅助组件并不是每次都要调用主 LLM 的全部能力，但又需要被频繁、低延迟地调用。

因此可以用蒸馏出来的小模型专门负责这类工作（管理记忆读写、检索判断、gate 决策等），让 O1（可插拔管理器）、O2（主 LLM）、O3（检索算法）等组件都可以在"小而专精"的前提下运行。

#### 3. 蒸馏于 RQ3 而言：蒸馏主要提升 G2（效率）

- **对 G1（回答准确度）**：蒸馏通常不带来提升，目标是保持准确度或将损失控制在可接受范围内（一定会有损失）。
- **对 G2（效率）**：这是蒸馏真正的价值所在。具体体现在：
    - **Loop efficiency**：小模型单轮速度远快于大模型，agent loop 更轻；
    - **KV cache 命中率**：参数化记忆本身就是"永久命中的缓存"；
    - **Memory budget**：把记忆操作蒸馏成 LoRA，降低了显式 memory 的存储预算。

#### 4. 蒸馏在 memory 语境下到底带来了什么？

**(a) 让 RQ2 中的辅助组件真的能在端侧跑起来**

- 云侧蒸馏：主要降成本（参数量、数据标注等）
- 端侧蒸馏：对我们意义重大。在端侧模型里编码用户身份与偏好，让随身的隐式记忆成为可能
- 参考：Apple 已在做端云混合架构，端侧 3B 处理日常（蒸馏 + LoRA 补精度），复杂任务走 Private Cloud Compute（[https://arxiv.org/abs/2507.13575](https://arxiv.org/abs/2507.13575)）
- 端侧算力与内存（如手机 16GB）对 3B 参数几乎是天花板，对端侧而言蒸馏几乎是必选项

**(b) Memory 场景永远在做一件事：把大的变小，但别丢信息**

- 长上下文要压缩
- 外部记忆库要缩减
- 向量检索要干掉（或至少减少依赖）
- 每一种"变小"都对应着一种技术路径，而蒸馏在这些路径上能很好地平衡压缩与保真。

**(c) 隐私**

- 端侧蒸馏 + 端侧推理：数据根本不出设备、压根不上传，隐私叙事天然成立
- 联邦蒸馏（federated distillation）是一个值得关注的方向：
    - 传统联邦学习的问题：传权重成本大（3B 模型梯度几十 GB，手机每天传不现实）；设备必须用同一个模型架构，否则梯度无法聚合
    - 联邦蒸馏只交换 logits（模型在一批公共数据上的输出分布，只有几 MB），服务器聚合 logits 做平均，再下发作为 soft label 继续训
    - 参考：[https://arxiv.org/abs/2304.01731](https://arxiv.org/abs/2304.01731)
